2746  /*
2747  Here is an outline of on-line/in-place ALTER TABLE execution through
2748  this interface.
2749 
2750  Phase 1 : Initialization
2751  ========================
2752  During this phase we determine which algorithm should be used
2753  for execution of ALTER TABLE and what level concurrency it will
2754  require.
2755 
2756  *) This phase starts by opening the table and preparing description
2757  of the new version of the table.
2758  *) Then we check if it is impossible even in theory to carry out
2759  this ALTER TABLE using the in-place algorithm. For example, because
2760  we need to change storage engine or the user has explicitly requested
2761  usage of the "copy" algorithm.
2762  *) If in-place ALTER TABLE is theoretically possible, we continue
2763  by compiling differences between old and new versions of the table
2764  in the form of HA_ALTER_FLAGS bitmap. We also build a few
2765  auxiliary structures describing requested changes and store
2766  all these data in the Alter_inplace_info object.
2767  *) Then the handler::check_if_supported_inplace_alter() method is called
2768  in order to find if the storage engine can carry out changes requested
2769  by this ALTER TABLE using the in-place algorithm. To determine this,
2770  the engine can rely on data in HA_ALTER_FLAGS/Alter_inplace_info
2771  passed to it as well as on its own checks. If the in-place algorithm
2772  can be used for this ALTER TABLE, the level of required concurrency for
2773  its execution is also returned.
2774  If any errors occur during the handler call, ALTER TABLE is aborted
2775  and no further handler functions are called.
2776  *) Locking requirements of the in-place algorithm are compared to any
2777  concurrency requirements specified by user. If there is a conflict
2778  between them, we either switch to the copy algorithm or emit an error.
2779 
2780  Phase 2 : Execution
2781  ===================
2782 
2783  In this phase the operations are executed.
2784 
2785  *) As the first step, we acquire a lock corresponding to the concurrency
2786  level which was returned by handler::check_if_supported_inplace_alter()
2787  and requested by the user. This lock is held for most of the
2788  duration of in-place ALTER (if HA_ALTER_INPLACE_SHARED_LOCK_AFTER_PREPARE
2789  or HA_ALTER_INPLACE_NO_LOCK_AFTER_PREPARE were returned we acquire an
2790  exclusive lock for duration of the next step only).
2791  *) After that we call handler::ha_prepare_inplace_alter_table() to give the
2792  storage engine a chance to update its internal structures with a higher
2793  lock level than the one that will be used for the main step of algorithm.
2794  After that we downgrade the lock if it is necessary.
2795  *) After that, the main step of this phase and algorithm is executed.
2796  We call the handler::ha_inplace_alter_table() method, which carries out the
2797  changes requested by ALTER TABLE but does not makes them visible to other
2798  connections yet.
2799  *) We ensure that no other connection uses the table by upgrading our
2800  lock on it to exclusive.
2801  *) a) If the previous step succeeds, handler::ha_commit_inplace_alter_table() is
2802  called to allow the storage engine to do any final updates to its structures,
2803  to make all earlier changes durable and visible to other connections.
2804  b) If we have failed to upgrade lock or any errors have occured during the
2805  handler functions calls (including commit), we call
2806  handler::ha_commit_inplace_alter_table()
2807  to rollback all changes which were done during previous steps.
2808 
2809  Phase 3 : Final
2810  ===============
2811 
2812  In this phase we:
2813 
2814  *) Update SQL-layer data-dictionary by installing .FRM file for the new version
2815  of the table.
2816  *) Inform the storage engine about this change by calling the
2817  handler::ha_notify_table_changed() method.
2818  *) Destroy the Alter_inplace_info and handler_ctx objects.
2819 
2820  */
