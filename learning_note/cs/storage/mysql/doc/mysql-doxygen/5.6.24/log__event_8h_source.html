98 /*
99  3 is MySQL 4.x; 4 is MySQL 5.0.0.
100  Compared to version 3, version 4 has:
101  - a different Start_log_event, which includes info about the binary log
102  (sizes of headers); this info is included for better compatibility if the
103  master's MySQL version is different from the slave's.
104  - all events have a unique ID (the triplet (server_id, timestamp at server
105  start, other) to be sure an event is not executed more than once in a
106  multimaster setup, example:
107  M1
108  / \
109  v v
110  M2 M3
111  \ /
112  v v
113  S
114  if a query is run on M1, it will arrive twice on S, so we need that S
115  remembers the last unique ID it has processed, to compare and know if the
116  event should be skipped or not. Example of ID: we already have the server id
117  (4 bytes), plus:
118  timestamp_when_the_master_started (4 bytes), a counter (a sequence number
119  which increments every time we write an event to the binlog) (3 bytes).
120  Q: how do we handle when the counter is overflowed and restarts from 0 ?
121 
122  - Query and Load (Create or Execute) events may have a more precise
123  timestamp (with microseconds), number of matched/affected/warnings rows
124  and fields of session variables: SQL_MODE,
125  FOREIGN_KEY_CHECKS, UNIQUE_CHECKS, SQL_AUTO_IS_NULL, the collations and
126  charsets, the PASSWORD() version (old/new/...).
127 */

217  MySQL Binary Log

219  This log consists of events. Each event has a fixed-length header,
220  possibly followed by a variable length data body.
221 
222  The data body consists of an optional fixed length segment (post-header)
223  and an optional variable length segment.
224

226

228  Execute_load_query_log_event and old Load_log_event and
229  Execute_load_log_event events (Execute_load_query is used together with
230  Begin_load_query and Append_block events to replicate LOAD DATA INFILE.
231  Create_file/Append_block/Execute_load (which includes Load_log_event)
232  were used to replicate LOAD DATA before the 5.0.3).

479  This flag only makes sense for Format_description_log_event. It is set
480  when the event is written, and *reset* when a binlog file is
481  closed (yes, it's the only case when MySQL modifies already written
482  part of binlog). Thus it is a reliable indicator that binlog was
483  closed correctly. (Stop_log_event is not enough, there's always a
484  small chance that mysqld crashes in the middle of insert and end of
485  the binlog would look like a Stop_log_event).
486
