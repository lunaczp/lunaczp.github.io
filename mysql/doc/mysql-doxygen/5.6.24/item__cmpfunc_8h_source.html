457 /*
458  trigcond<param>(arg) ::= param? arg : TRUE
459 
460  The class Item_func_trig_cond is used for guarded predicates
461  which are employed only for internal purposes.
462  A guarded predicate is an object consisting of an a regular or
463  a guarded predicate P and a pointer to a boolean guard variable g.
464  A guarded predicate P/g is evaluated to true if the value of the
465  guard g is false, otherwise it is evaluated to the same value that
466  the predicate P: val(P/g)= g ? val(P):true.
467  Guarded predicates allow us to include predicates into a conjunction
468  conditionally. Currently they are utilized for pushed down predicates
469  in queries with outer join operations.
470 
471  In the future, probably, it makes sense to extend this class to
472  the objects consisting of three elements: a predicate P, a pointer
473  to a variable g and a firing value s with following evaluation
474  rule: val(P/g,s)= g==s? val(P) : true. It will allow us to build only
475  one item for the objects of the form P/g1/g2...
476 
477  Objects of this class are built only for query execution after
478  the execution plan has been already selected. That's why this
479  class needs only val_int out of generic methods.
480 
481  Current uses of Item_func_trig_cond objects:
482  - To wrap selection conditions when executing outer joins
483  - To wrap condition that is pushed down into subquery
484 */

1705 /*
1706  The class Item_equal is used to represent conjunctions of equality
1707  predicates of the form field1 = field2, and field=const in where
1708  conditions and on expressions.
1709 
1710  All equality predicates of the form field1=field2 contained in a
1711  conjunction are substituted for a sequence of items of this class.
1712  An item of this class Item_equal(f1,f2,...fk) represents a
1713  multiple equality f1=f2=...=fk.
1714 
1715  If a conjunction contains predicates f1=f2 and f2=f3, a new item of
1716  this class is created Item_equal(f1,f2,f3) representing the multiple
1717  equality f1=f2=f3 that substitutes the above equality predicates in
1718  the conjunction.
1719  A conjunction of the predicates f2=f1 and f3=f1 and f3=f2 will be
1720  substituted for the item representing the same multiple equality
1721  f1=f2=f3.
1722  An item Item_equal(f1,f2) can appear instead of a conjunction of
1723  f2=f1 and f1=f2, or instead of just the predicate f1=f2.
1724 
1725  An item of the class Item_equal inherits equalities from outer
1726  conjunctive levels.
1727 
1728  Suppose we have a where condition of the following form:
1729  WHERE f1=f2 AND f3=f4 AND f3=f5 AND ... AND (...OR (f1=f3 AND ...)).
1730  In this case:
1731  f1=f2 will be substituted for Item_equal(f1,f2);
1732  f3=f4 and f3=f5 will be substituted for Item_equal(f3,f4,f5);
1733  f1=f3 will be substituted for Item_equal(f1,f2,f3,f4,f5);
1734 
1735  An object of the class Item_equal can contain an optional constant
1736  item c. Then it represents a multiple equality of the form
1737  c=f1=...=fk.
1738 
1739  Objects of the class Item_equal are used for the following:
1740 
1741  1. An object Item_equal(t1.f1,...,tk.fk) allows us to consider any
1742  pair of tables ti and tj as joined by an equi-condition.
1743  Thus it provide us with additional access paths from table to table.
1744 
1745  2. An object Item_equal(t1.f1,...,tk.fk) is applied to deduce new
1746  SARGable predicates:
1747  f1=...=fk AND P(fi) => f1=...=fk AND P(fi) AND P(fj).
1748  It also can give us additional index scans and can allow us to
1749  improve selectivity estimates.
1750 
1751  3. An object Item_equal(t1.f1,...,tk.fk) is used to optimize the
1752  selected execution plan for the query: if table ti is accessed
1753  before the table tj then in any predicate P in the where condition
1754  the occurrence of tj.fj is substituted for ti.fi. This can allow
1755  an evaluation of the predicate at an earlier step.
1756 
1757  When feature 1 is supported they say that join transitive closure
1758  is employed.
1759  When feature 2 is supported they say that search argument transitive
1760  closure is employed.
1761  Both features are usually supported by preprocessing original query and
1762  adding additional predicates.
1763  We do not just add predicates, we rather dynamically replace some
1764  predicates that can not be used to access tables in the investigated
1765  plan for those, obtained by substitution of some fields for equal fields,
1766  that can be used.
1767 
1768  Prepared Statements/Stored Procedures note: instances of class
1769  Item_equal are created only at the time a PS/SP is executed and
1770  are deleted in the end of execution. All changes made to these
1771  objects need not be registered in the list of changes of the parse
1772  tree and do not harm PS/SP re-execution.
1773 
1774  Item equal objects are employed only at the optimize phase. Usually they are
1775  not supposed to be evaluated. Yet in some cases we call the method val_int()
1776  for them. We have to take care of restricting the predicate such an
1777  object represents f1=f2= ...=fn to the projection of known fields fi1=...=fik.
1778 */
