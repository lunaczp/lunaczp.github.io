57  {
58  /*
59  About HA_FAST_KEY_READ:
60 
61  The storage engine ::rnd_pos() method is fast to locate records by key,
62  so HA_FAST_KEY_READ is technically true, but the record content can be
63  overwritten between ::rnd_next() and ::rnd_pos(), because all the P_S
64  data is volatile.
65  The HA_FAST_KEY_READ flag is not advertised, to force the optimizer
66  to cache records instead, to provide more consistent records.
67  For example, consider the following statement:
68  - select * from P_S.EVENTS_WAITS_HISTORY_LONG where THREAD_ID=<n>
69  order by ...
70  With HA_FAST_KEY_READ, it can return records where "THREAD_ID=<n>"
71  is false, because the where clause was evaluated to true after
72  ::rnd_pos(), then the content changed, then the record was fetched by
73  key using ::rnd_pos().
74  Without HA_FAST_KEY_READ, the optimizer reads all columns and never
75  calls ::rnd_pos(), so it is guaranteed to return only thread <n>
76  records.
77  */
78  return HA_NO_TRANSACTIONS | HA_REC_NOT_IN_SEQ | HA_NO_AUTO_INCREMENT |
79  HA_PRIMARY_KEY_REQUIRED_FOR_DELETE;
80  }
