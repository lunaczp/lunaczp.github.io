70 /*
71  JOIN_CACHE is the base class to support the implementations of both
72  Blocked-Based Nested Loops (BNL) Join Algorithm and Batched Key Access (BKA)
73  Join Algorithm. The first algorithm is supported by the derived class
74  JOIN_CACHE_BNL, while the second algorithm is supported by the derived
75  class JOIN_CACHE_BKA.
76  These two algorithms have a lot in common. Both algorithms first
77  accumulate the records of the left join operand in a join buffer and
78  then search for matching rows of the second operand for all accumulated
79  records.
80  For the first algorithm this strategy saves on logical I/O operations:
81  the entire set of records from the join buffer requires only one look-through
82  the records provided by the second operand.
83  For the second algorithm the accumulation of records allows to optimize
84  fetching rows of the second operand from disk for some engines (MyISAM,
85  InnoDB), or to minimize the number of round-trips between the Server and
86  the engine nodes (NDB Cluster).
87 */

592 /*
593  The class JOIN_CACHE_BKA_UNIQUE supports the variant of the BKA join algorithm
594  that submits only distinct keys to the MRR interface. The records in the join
595  buffer of a cache of this class that have the same access key are linked into
596  a chain attached to a key entry structure that either itself contains the key
597  value, or, in the case when the keys are embedded, refers to its occurance in
598  one of the records from the chain.
599  To build the chains with the same keys a hash table is employed. It is placed
600  at the very end of the join buffer. The array of hash entries is allocated
601  first at the very bottom of the join buffer, then go key entries. A hash entry
602  contains a header of the list of the key entries with the same hash value.
603  Each key entry is a structure of the following type:
604  struct st_join_cache_key_entry {
605  union {
606  uchar[] value;
607  cache_ref *value_ref; // offset from the beginning of the buffer
608  } hash_table_key;
609  key_ref next_key; // offset backward from the beginning of hash table
610  cache_ref *last_rec // offset from the beginning of the buffer
611  }
612  The references linking the records in a chain are always placed at the very
613  beginning of the record info stored in the join buffer. The records are
614  linked in a circular list. A new record is always added to the end of this
615  list. When a key is passed to the MRR interface it can be passed either with
616  an association link containing a reference to the header of the record chain
617  attached to the corresponding key entry in the hash table, or without any
618  association link. When the next record is returned by a call to the MRR
619  function multi_range_read_next without any association (because if was not
620  passed together with the key) then the key value is extracted from the
621  returned record and searched for it in the hash table. If there is any records
622  with such key the chain of them will be yielded as the result of this search.
623 
624  The following picture represents a typical layout for the info stored in the
625  join buffer of a join cache object of the JOIN_CACHE_BKA_UNIQUE class.
626 
627  buff
628  V
629  +----------------------------------------------------------------------------+
630  | |[*]record_1_1| |
631  | ^ | |
632  | | +--------------------------------------------------+ |
633  | | |[*]record_2_1| | |
634  | | ^ | V |
635  | | | +------------------+ |[*]record_1_2| |
636  | | +--------------------+-+ | |
637  |+--+ +---------------------+ | | +-------------+ |
638  || | | V | | |
639  |||[*]record_3_1| |[*]record_1_3| |[*]record_2_2| | |
640  ||^ ^ ^ | |
641  ||+----------+ | | | |
642  ||^ | |<---------------------------+-------------------+ |
643  |++ | | ... mrr | buffer ... ... | | |
644  | | | | |
645  | +-----+--------+ | +-----|-------+ |
646  | V | | | V | | |
647  ||key_3|[/]|[*]| | | |key_2|[/]|[*]| | |
648  | +-+---|-----------------------+ | |
649  | V | | | | |
650  | |key_1|[*]|[*]| | | ... |[*]| ... |[*]| ... | |
651  +----------------------------------------------------------------------------+
652  ^ ^ ^
653  | i-th entry j-th entry
654  hash table
655 
656  i-th hash entry:
657  circular record chain for key_1:
658  record_1_1
659  record_1_2
660  record_1_3 (points to record_1_1)
661  circular record chain for key_3:
662  record_3_1 (points to itself)
663 
664  j-th hash entry:
665  circular record chain for key_2:
666  record_2_1
667  record_2_2 (points to record_2_1)
668 
669 */
